= Repository Guide

This guide is the fastest way to understand and work inside this monorepo.
Use it as the onboarding index before diving into module-specific READMEs.

== What this repository contains

This repository hosts the tBTC v2 protocol codebase and related tooling:

- Core protocol smart contracts and deployments
- Cross-chain contract variants
- TypeScript SDK
- System and monitoring tools
- Operator runbooks and starter kits
- pBTC demo applications (frontend + bridge API)

== Start here by goal

=== I want to work on core protocol contracts

- Start with link:../solidity/README.adoc[solidity/README.adoc]
- Primary commands:
  - `yarn install`
  - `yarn build`
  - `yarn test`
  - `yarn test:integration`

=== I want to work on the SDK

- Start with link:../typescript/README.md[typescript/README.md]

== Environment and toolchain

Because this is a long-lived monorepo, modules can have different Node expectations.
Always defer to the module README first.

Baseline tools you will commonly need:

- Node.js (module-dependent; several legacy modules still reference Node 14+)
- Yarn classic (`1.x`) for most legacy modules
- npm (used by `apps/pbtc-bridge-api`)
- Docker (for some operator/monitoring workflows)

Repository baseline:

- Root `.nvmrc` uses `lts/hydrogen` as a default baseline.
- Some modules require older/newer Node versions; follow module README first.

Quick bootstrap:

- Run `npm run dev:bootstrap` at repository root for a module command matrix.

Potential Git dependency workaround (applies to some legacy transitive dependencies):

`git config --global url."https://".insteadOf git://`

== Common workflows

=== Formatting (repo root)

At repository root:

- `npm run format`
- `npm run format:fix`

=== Contract smoke CI for bridge API

Bridge API contract smoke tests are executed by:

- local: `npm run test:contract` in `apps/pbtc-bridge-api`
- CI workflow: `pbtc-bridge-api.yml`

=== Typical change strategy

1. Make targeted changes in one module.
2. Run module-local build/test commands first.
3. Run broader checks only if your change spans modules.
